# 测试策略

本文定义 HorrorCooked9 的最低测试要求与长期目标，为开发者提供可执行的质量基线。

## 1. 当前最低要求

- **编译通过**：每次 PR 必须通过 `gradlew build`（由 CI 自动执行）。
- **手动冒烟测试**：涉及玩法逻辑变更时，开发者应在本地启动游戏进行基本功能验证，确认关键路径无崩溃。

## 2. 建议逐步引入

| 测试类型 | 适用范围 | 工具 | 优先级 |
| --- | --- | --- | --- |
| 纯 Java 单元测试 | 配置解析、数据驱动配方匹配、工具函数 | JUnit 5（`src/test/java`） | P1 |
| Forge GameTest | 方块交互、配方流程、网络包往返 | `@GameTest` 注解 + `gameTestServer` 运行配置 | P2 |
| 集成 / 回归测试 | 跨版本存档兼容、数据包热替换 | 手动或脚本化启动流程 | P3 |

## 3. 测试命名与目录

- 单元测试放在 `src/test/java/net/qiuyu/horrorcooked9/`，包结构与主代码对齐。
- GameTest 放在 `src/main/java` 对应包下，使用 `@GameTest` 注解，并在 `build.gradle` 的 `gameTestServer` 运行配置中注册。
- 测试类以 `*Test` 结尾。

## 4. 何时必须写测试

当前阶段不做强制要求，但以下场景强烈建议补测试：

- 新增或修改配置解析逻辑（如 `FoodRuntimeConfigs` 的 JSON 字段变更）。
- 修复影响玩法结果的 bug（回归测试）。
- 跨层桥接路径变更（确保服务端与客户端行为一致）。
